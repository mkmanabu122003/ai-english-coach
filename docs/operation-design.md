# AI English Coach Bot 運用設計書

通訳ガイド向け英語コーチングスクール生徒用 LINE Bot の運用設計

**現在の規模**: 生徒2名
**目標規模**: 数百名
**対応言語**: 英語（初期）→ スペイン語（将来拡張）

---

## 1. コンセプトと設計思想

### 1.1 Botの位置づけ

```
対面レッスン（週1回）─── 知識のインプット・深い学び
        ↕
   AI English Coach Bot ─── 日常の反復練習・アウトプット習慣化
        ↕
   講師ダッシュボード ──── 生徒別プラン管理・進捗モニタリング
```

Botは**対面レッスンの補助ツール**であり、代替ではない。
目的は「毎日少しでも英語に触れる習慣」をつくり、対面レッスンの効果を最大化すること。

### 1.2 コア方針: "comparison, not translation"

通訳ガイドの現場で最も求められるのは「日本文化を外国人に分かりやすく伝える」スキル。
直訳ではなく、相手の文化圏の概念と比較して説明する方法を一貫して訓練する。

### 1.3 スケーラビリティ方針

2名→数百名への成長を見据え、以下を設計原則とする：

- **生徒別プラン管理**: 各生徒に学習プランを割り当て、進捗を個別追跡
- **グループ/クラス管理**: 講師が生徒をグループ化し、一括管理可能に
- **多言語対応**: 英語版をベースにスペイン語版を展開可能なアーキテクチャ
- **コスト最適化**: 人数増に伴うAPI費用・LINE送信費用の管理

## 2. マーケティング戦略

### 2.1 ターゲットペルソナ定義（MA-1）

「通訳ガイド向け」の中でも、ターゲットによってアプローチが大きく異なる。以下の3ペルソナを定義し、メッセージングと導入チャネルを分ける。

#### ペルソナ A: 現役フルタイムガイド（30-50代）

| 項目 | 内容 |
|------|------|
| 属性 | 全国通訳案内士資格保持、年間100件以上のツアー実績 |
| 課題 | 表現がパターン化し成長実感がない、音声での即興対応に苦手意識 |
| 動機 | スキルアップによる単価向上、リピート客の獲得 |
| チャネル | 業界団体・ガイド協会への直接アプローチ、口コミ |
| Bot利用傾向 | 音声練習中心、Advanced レベル、短時間で高密度な練習を好む |

#### ペルソナ B: 資格取得を目指す学習者（20-40代）

| 項目 | 内容 |
|------|------|
| 属性 | 英語力はTOEIC600-800程度、通訳ガイド試験の準備中 |
| 課題 | 観光英語の実践経験がない、日本文化の英語説明に不慣れ |
| 動機 | 試験対策＋実務準備を同時に進めたい |
| チャネル | 通訳ガイド養成学校との提携、SNS広告 |
| Bot利用傾向 | テキスト中心、Beginner〜Intermediate、毎日の習慣化に意欲的 |

#### ペルソナ C: 副業・週末ガイド / 観光ボランティア（40-60代）

| 項目 | 内容 |
|------|------|
| 属性 | 地域の観光ボランティアや週末のみのガイド活動 |
| 課題 | デジタルツールへの不慣れ、学習時間の確保が難しい |
| 動機 | 地域の魅力を外国人に伝えたい、社会貢献 |
| チャネル | 地方自治体の観光課、地域ガイド団体 |
| Bot利用傾向 | テキスト中心、Beginner、丁寧なUI・操作ガイドが必要 |

### 2.2 集客・新規獲得戦略（MA-2）

2名→数百名へスケールするための段階的な集客チャネル設計。

#### フェーズ別集客チャネル

| フェーズ | 目標人数 | 主要チャネル | 施策 |
|---------|---------|------------|------|
| Phase 1 | 2→10名 | 口コミ・紹介 | 既存生徒の紹介インセンティブ、講師の人脈活用 |
| Phase 2 | 10→50名 | 提携・SNS | 養成学校への提案、Instagram/X でのコンテンツ発信 |
| Phase 3 | 50→300名 | 法人・広告 | 旅行会社/自治体への法人営業、リスティング広告 |

#### 紹介プログラム

```
既存生徒が新規生徒を紹介
  → 紹介者: 翌月のBot利用料 50% OFF or 追加音声練習5回/日に拡張
  → 被紹介者: 初月無料 or 14日間無料トライアル
```

#### コンテンツマーケティング

- Bot の質問・模範回答例をSNSで定期発信（週2-3回）
- 「通訳ガイドの英語Tips」としてブランド認知を拡大
- ハッシュタグ: #通訳ガイド英語 #AI英語コーチ #GuideEnglish

#### 無料トライアル設計

```
7日間無料トライアル:
  - プッシュ通知: 毎日1問
  - テキスト練習: 3回/日
  - 音声練習: 1回/日
  - ウィークリーレポート: 1回

トライアル終了時:
  → 自動で利用停止メッセージ + 有料プランへの案内
  → 3日後にフォローアップメッセージ（トライアル中の成果を振り返り）
  → 講師が対面レッスンで直接コンバージョン
```

## 5. フェーズ別運用設計

### 5.1 導入期（Week 0）: オンボーディング

**目標**: 初日に「Bot返信の体験」を完了させ、翌日プッシュ通知の受け取りまでつなげる

#### 現状のウェルカムメッセージ
```
こんにちは！AI English Coachです 🎓
英文を送ると添削します。音声メッセージもOKです。
まずはこれを英語で答えてみてください👇
「外国人観光客に鳥居を説明するとしたら、どう言いますか？」
```

#### 改善案: 3ステップ・オンボーディング

**ステップ1: フォロー直後（即時）**
```
はじめまして！AI English Coach です 🎓
通訳ガイドの英語力アップをお手伝いします。

このBotでできること：
📝 英文を送ると → 添削＋ガイド向け表現を提案
🎤 音声を送ると → 発音チェック＋改善ポイント
📊 毎週日曜に → 学習レポートをお届け

さっそく始めましょう！
まずは英語であなたの自己紹介をしてください。
例: "Hi, I'm [名前]. I've been a tour guide for [X] years."
```

> **設計意図**: 自己紹介はどのレベルでも答えられる。かつ内容の充実度でレベル判定が正確になる。

**ステップ2: 初回返信後（AIレスポンスの末尾に追加）**
```
[通常のフィードバック]

──────
💡 ヒント: 毎朝、今日の練習問題が届きます。
通知時間の変更は「通知設定」と送ってください。
```

**ステップ3: 翌日のプッシュ通知（初回専用メッセージ）**

```
おはようございます！昨日の自己紹介、素晴らしかったです ✨

🗣️ [質問]
```

#### 講師側の運用

### 5.2 習慣形成期（Week 1-2）: プッシュ通知の最適化

## 6. カスタマーサクセス戦略

### 6.1 チャーン（離脱）防止フロー（CS-1）

ナッジメッセージに加え、段階的な介入フローを設計する。

#### 離脱リスクスコアリング

| シグナル | スコア加算 | 説明 |
|---------|----------|------|
| 1日無活動 | +1 | 通常範囲 |
| 3日連続無活動 | +3 | 要注意 |
| ストリーク途切れ | +2 | モチベーション低下の兆候 |
| 応答文字数の減少傾向 | +2 | エンゲージメント低下 |
| 音声利用の停止 | +1 | 練習の質低下 |
| ウィークリーレポート未読 | +1 | 関心低下 |

リスクスコアが **5以上** で「要フォロー」、**8以上** で「要介入」と判定。

#### 段階的介入フロー

```
┌─────────────────────────────────────────────────────────┐
│  無活動日数    │  アクション                              │
├─────────────────────────────────────────────────────────┤
│  2日          │ Bot自動ナッジ（gentle）                   │
│  3日          │ Bot自動ナッジ（strong）+ ストリーク復活案内 │
│  5日          │ 講師にLINEアラート通知                     │
│               │ 「⚠️ 山田さんが5日間無活動です」           │
│  7日          │ 講師から個別メッセージ送信                  │
│               │ 「最近忙しいですか？5分だけでも…」          │
│  10日         │ 講師から直接連絡（LINE通話 or 対面時に声がけ）│
│  30日         │ 休会提案 or 退会フォロー                   │
│               │ 休会→1ヶ月後にリカバリーメッセージ          │
└─────────────────────────────────────────────────────────┘
```

#### 講師アラート通知の実装

```
Firestore: alerts/
  {alertId}/
    type: "inactive_student"
    studentId: "U..."
    studentName: "山田さん"
    inactiveDays: 5
    lastActiveDate: "2026-03-08"
    status: "pending" | "acknowledged" | "resolved"
    createdAt: Timestamp

→ Cloud Functions で毎朝チェック、閾値超過時にLINEで講師に通知
```

#### 離脱理由の収集

- LINE上でのブロック/unfollowは検知不可のため、対面レッスン時にヒアリング
- 休会/退会フォーム（LINE内コマンド「退会理由」で簡易アンケート送信）
- 選択肢: 忙しい / 難しすぎる / 簡単すぎる / 対面だけで十分 / その他

```
  {lineUserId}/                      ← メンバー1, メンバー2...
    role: "member" | "admin"
    enrolledAt: Timestamp
    status: "active" | "paused"

users/{lineUserId}/                  ← 既存のUserコレクション（変更最小限）
```

> 個人ユーザーと法人メンバーが共存可能。破壊的変更なし。

## 8. 多言語対応設計（スペイン語版）

### 8.1 アーキテクチャ方針

**同一のLINE公式アカウントで複数言語に対応**するか、**言語別にアカウントを分ける**かの選択。

#### 推奨: 言語別LINE公式アカウント

| 方式 | メリット | デメリット |
|------|---------|----------|
| 同一アカウント | 管理が1つ、コード共有 | ユーザー体験が複雑、プロンプト混在 |
| **別アカウント** | **ユーザー体験がクリーン、独立運用** | アカウント2つの管理 |

→ 言語ごとにLINE公式アカウントを作成し、**バックエンドコードは共有**する。

### 8.2 コード上の多言語対応

```
src/
  prompts/
    systemPrompt.ts      → 言語パラメータを受け取るように変更
    voicePrompt.ts       → 同上
    reportPrompt.ts      → 同上
    pushMessages/
      en.ts              → 英語版質問プール
      es.ts              → スペイン語版質問プール
  config/
    constants.ts         → 言語別設定
    languages.ts         → 言語定義・UIテキスト
```

### 8.3 User型の拡張

```typescript
export interface User {
  // ... 既存フィールド ...
  language: "en" | "es";        // 学習対象言語
  uiLanguage: "ja";             // UI言語（当面は日本語固定）
  planId: string;               // 所属プランID
  courseId: string;              // 所属コースID
}
```

### 8.4 システムプロンプトの多言語化

```
英語版:
"You are an AI English coach specialized in tourism English
 for Japanese tour guides..."

スペイン語版:
"You are an AI Spanish coach specialized in tourism Spanish
 for Japanese tour guides. Your role is to help Japanese guides
 explain Japanese culture to Spanish-speaking tourists..."
```

**スペイン語版の特徴**:
- 比較対象がスペイン・中南米文化に変わる

  - おもてなし → "Como la hospitalidad mediterránea, pero anticipando necesidades"
- Speech-to-Text の言語設定を `es-ES` に変更
- 質問プールをスペイン語圏の観光客向けに調整

### 8.5 多言語展開のタイムライン

> **TM-4 協議事項**: 多言語対応の実装方式について、以下の2案を検討し決定する必要がある。

## 10. レート制限と利用ガイドライン

### 現在の制限値

| 項目 | 制限 | 根拠 |
|------|------|------|
| テキスト | 20回/日 | 1回あたり約800トークン × 20 = API費用抑制 |
| 音声 | 5回/日 | Speech-to-Text + Claude のダブルAPI費用 |
| 音声長さ | 180秒/回 | Speech-to-Text の精度維持 |
| コンテキスト | 直近5件 or 60分 | トークン費用と文脈精度のバランス |

### 推奨利用パターン

```
【1日の理想的な利用】 所要時間: 10-15分

1. プッシュ通知の質問に英語で回答（テキスト）     ... 3-5分
2. AIフィードバックを読んで修正版を送信（テキスト） ... 2-3分
3. 修正版を音声で言ってみる（音声）              ... 2-3分
4. （余裕があれば）自分で考えた表現を送る         ... 3-5分

→ テキスト3-4回 + 音声1回 = 合計4-5回/日
→ 制限の25%程度しか使わないので十分な余裕
```

### スクール内でのルール案

1. **最低ライン**: 1日1回、プッシュ通知に返信する
2. **推奨ライン**: 上記パターンで1日10-15分
3. **ストリーク目標**: まず7日連続、次に30日連続を目指す
4. **音声の重要性**: 週に最低3回は音声メッセージで練習する

```
プロンプト変更時に出力品質が劣化していないかを自動チェック。

手順:
  1. テスト入力セット（各レベル×各カテゴリ = 18パターン）を用意
  2. 変更前プロンプトで出力を生成・保存（ゴールデンファイル）
  3. 変更後プロンプトで出力を生成
  4. 以下の観点で自動比較:
     - フィードバックの構造（改善ポイント数、文化比較の有無）
     - 出力トークン数の大幅な増減（±30%以上で警告）
     - 日本語UIテキストの崩れ
  5. 差分が大きい場合は人間がレビュー
```

### 14.3 環境分離（TM-3）

プロンプト変更やコード変更が直接生徒に影響しないよう、環境を分離する。

#### 環境構成

| 環境 | 用途 | Firebase プロジェクト | LINE アカウント |
|------|------|---------------------|----------------|
| **開発** | ローカル開発・単体テスト | Firebase Emulators | 不要 |
| **ステージング** | 統合テスト・プロンプト検証 | 別プロジェクト（`xxx-staging`） | テスト用公式アカウント |
| **本番** | 実際の生徒が利用 | 本番プロジェクト | 本番公式アカウント |

#### デプロイフロー

```
feature branch → PR → ステージング自動デプロイ
  → 講師がテスト生徒としてBot動作確認
  → 問題なければ main にマージ
  → 本番自動デプロイ

プロンプト変更の追加ステップ:
  → ステージングでプロンプト回帰テスト実行
  → テスト入力に対する出力品質を講師が確認
  → 承認後に本番マージ
```

#### 環境変数の管理

```
.env.development   → Firebase Emulators 接続情報
.env.staging        → ステージング Firebase + テスト LINE
.env.production     → 本番 Firebase + 本番 LINE

機密情報: Firebase Secret Manager で管理
  - ANTHROPIC_API_KEY
  - LINE_CHANNEL_SECRET
  - LINE_CHANNEL_ACCESS_TOKEN
  - GOOGLE_CLOUD_API_KEY
```

### 14.4 データバックアップ・災害復旧計画（TM-5）

#### バックアップ設計

| 対象 | 方式 | 頻度 | 保持期間 |
|------|------|------|---------|
| Firestore全データ | `gcloud firestore export` | 日次（深夜3時） | 30日間 |
| Cloud Functions コード | Git リポジトリ | コミット時 | 永続 |
| 環境変数・シークレット | Secret Manager バージョニング | 変更時 | 全バージョン |
| チャットログ | Firestore バックアップに含む | 日次 | 30日間 |

#### バックアップ自動化

```
Cloud Scheduler（毎日 03:00 JST）
  → Cloud Functions: backupFirestore
  → gcloud firestore export gs://xxx-backups/$(date)
  → 完了通知: Slack

30日超過バックアップの自動削除:
  → Cloud Functions: cleanupOldBackups（週次実行）
```

#### 災害復旧手順

```
リージョン障害時（asia-northeast1）:
  1. GCPステータスページで障害状況を確認
  2. 復旧見込みが24時間以上の場合:
     a. 別リージョン（asia-northeast2）に新プロジェクト作成
     b. 最新バックアップからFirestoreリストア
     c. Cloud Functions を再デプロイ
     d. LINE Webhook URLを新エンドポイントに変更
  3. 生徒への影響: プッシュ通知の一時停止、チャット応答不可
  4. 復旧後: データ整合性チェック、ストリーク補正

RTO（目標復旧時間）: 4時間
RPO（目標復旧時点）: 24時間（日次バックアップ間隔）
```

### 14.5 AIハルシネーション対策（TM-6）

通訳ガイド向けという性質上、文化・歴史的に不正確な情報を生徒に教えるリスクがある。

#### プロンプトレベルの対策

```
システムプロンプトに追加する指示:
  - 「文化的・歴史的事実について確信がない場合は断定を避け、
    "This is one interpretation..." 等の表現を使うこと」
  - 「年号、人名、地名の正確性に特に注意すること」
  - 「生徒の回答に含まれる事実誤認を指摘する際は、
    正確な情報源を示唆すること（例: "You might want to check..."）」
```

#### 報告・検証の仕組み

```
生徒からの報告コマンド:
  「間違い報告」と送信 → 直前のAIフィードバックにフラグを立てる

Firestore: errorReports/
  {reportId}/
    userId: "U..."
    chatId: "chat_xxx"
    aiResponse: "（問題のAI回答）"
    status: "reported" | "reviewing" | "confirmed" | "dismissed"
    reviewedBy: "instructor_001"
    createdAt: Timestamp

講師の確認フロー:
  1. 週次で errorReports を確認
  2. 事実誤認が確認された場合:
     a. 生徒に訂正メッセージを送信
     b. プロンプトの改善を検討
     c. 類似エラーの再発防止策を記録
```

#### 高リスクカテゴリの特別対応

| カテゴリ | リスク | 対策 |
|---------|-------|------|
| history | 年号・人物の誤り | 主要事実のファクトチェックリスト作成 |
| shrine_temple | 宗教的な誤解 | 神道/仏教の基本事実をプロンプトに明記 |
| concept | 文化概念の過度な単純化 | 「複数の解釈がある」旨を付記する指示 |
| food | アレルギー・宗教的食事制限 | 安全に関わる情報は慎重に扱う指示 |
