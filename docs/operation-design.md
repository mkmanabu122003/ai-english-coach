# AI English Coach Bot 運用設計書

通訳ガイド向け英語コーチングスクール生徒用 LINE Bot の運用設計

**現在の規模**: 生徒2名
**目標規模**: 数百名
**対応言語**: 英語（初期）→ スペイン語（将来拡張）

---

## 1. コンセプトと設計思想

### 1.1 Botの位置づけ

```
対面レッスン（週1回）─── 知識のインプット・深い学び
        ↕
   AI English Coach Bot ─── 日常の反復練習・アウトプット習慣化
        ↕
   講師ダッシュボード ──── 生徒別プラン管理・進捗モニタリング
```

Botは**対面レッスンの補助ツール**であり、代替ではない。
目的は「毎日少しでも英語に触れる習慣」をつくり、対面レッスンの効果を最大化すること。

### 1.2 コア方針: "comparison, not translation"

通訳ガイドの現場で最も求められるのは「日本文化を外国人に分かりやすく伝える」スキル。
直訳ではなく、相手の文化圏の概念と比較して説明する方法を一貫して訓練する。

### 1.3 スケーラビリティ方針

2名→数百名への成長を見据え、以下を設計原則とする：

- **生徒別プラン管理**: 各生徒に学習プランを割り当て、進捗を個別追跡
- **グループ/クラス管理**: 講師が生徒をグループ化し、一括管理可能に
- **多言語対応**: 英語版をベースにスペイン語版を展開可能なアーキテクチャ
- **コスト最適化**: 人数増に伴うAPI費用・LINE送信費用の管理

## 3. フェーズ別運用設計

### 3.1 導入期（Week 0）: オンボーディング

**目標**: 初日に「Bot返信の体験」を完了させ、翌日プッシュ通知の受け取りまでつなげる

#### 現状のウェルカムメッセージ
```
こんにちは！AI English Coachです 🎓
英文を送ると添削します。音声メッセージもOKです。
まずはこれを英語で答えてみてください👇
「外国人観光客に鳥居を説明するとしたら、どう言いますか？」
```

#### 改善案: 3ステップ・オンボーディング

**ステップ1: フォロー直後（即時）**
```
はじめまして！AI English Coach です 🎓
通訳ガイドの英語力アップをお手伝いします。

このBotでできること：
📝 英文を送ると → 添削＋ガイド向け表現を提案
🎤 音声を送ると → 発音チェック＋改善ポイント
📊 毎週日曜に → 学習レポートをお届け

さっそく始めましょう！
まずは英語であなたの自己紹介をしてください。
例: "Hi, I'm [名前]. I've been a tour guide for [X] years."
```

> **設計意図**: 自己紹介はどのレベルでも答えられる。かつ内容の充実度でレベル判定が正確になる。

**ステップ2: 初回返信後（AIレスポンスの末尾に追加）**
```
[通常のフィードバック]

──────
💡 ヒント: 毎朝、今日の練習問題が届きます。
通知時間の変更は「通知設定」と送ってください。
```

**ステップ3: 翌日のプッシュ通知（初回専用メッセージ）**
```
おはようございます！昨日の自己紹介、素晴らしかったです ✨
今日は実践的な場面にチャレンジしてみましょう👇

🗣️ [質問]
```

#### 講師側の運用
- 生徒に**対面レッスン時にQRコードを読み取ってもらう**（友だち追加）
- Bot追加後、その場で1通目を送る体験をさせる
- 翌日の通知が届くことを伝え、「まず1週間毎日1問やってみて」と声がけ

---

### 3.2 習慣形成期（Week 1-2）: プッシュ通知の最適化

**目標**: 毎日1回Botに返信する習慣をつける

#### プッシュ通知のタイミング設計

| 時間帯 | 推奨対象 | 理由 |
|--------|---------|------|
| 07:00-08:00 | 朝型の生徒 | 通勤前の隙間時間 |
| 12:00-13:00 | 日中の生徒 | 昼休み |
| 21:00-22:00 | 夜型の生徒 | 就寝前の学習 |

**改善提案**: 通知時間を生徒が自分で設定できるようにする（現状pushTimeフィールドはあるが変更UIがない）

```
コマンド追加案:
「通知設定 08:00」→ プッシュ通知時間を08:00に変更
「通知設定 21:00」→ プッシュ通知時間を21:00に変更
```

#### レベル別質問の出し分け（新機能）

現在30問の質問プールは全レベル共通。レベル別に適切な難易度で出し分ける。

| レベル | 質問の特徴 | 例 |
|--------|-----------|-----|
| Beginner | 短い回答で済む、具体的な場面 | "How do you say 'いただきます' to your guests at lunch?" |
| Intermediate | 比較・説明が必要 | "How would you explain the difference between a shrine and a temple?" |
| Advanced | 深い文化理解＋即興力 | "A guest challenges you: 'Isn't wabi-sabi just an excuse for imperfection?' How do you respond?" |

---

### 3.3 成長期（Week 3-8）: 対面レッスンとの連携

**目標**: Botでの自主練習と対面レッスンが相互に強化しあう

#### 週間サイクルの設計

```
月    ─ プッシュ通知（今週のテーマ導入）
火-金 ─ 毎日1問テキストor音声で回答
土    ─ 対面レッスン（今週の質問をもとにロールプレイ）
日    ─ ウィークリーレポート配信
```

#### 「今週のテーマ」制（新機能）

1週間同じカテゴリの質問を集中的に出すことで、語彙と表現パターンの定着を促進。

```
Week 1: shrine_temple（神社仏閣）
Week 2: food（食文化）
Week 3: daily_life（日常生活）
Week 4: concept（日本文化の概念）
Week 5: practical（実践的な場面対応）
Week 6: nature（自然・季節）
Week 7: shrine_temple（2周目 - より深い質問）
...
```

**講師側の運用**:
- 対面レッスンのテーマを週テーマと合わせる
- 「今週Botでやった質問の中で難しかったのは？」からレッスンを始める
- 生徒のウィークリーレポートを事前に確認して指導に活かす

#### 音声練習の積極活用

通訳ガイドは**話す力**が最重要。テキストだけでなく音声練習を促す仕組み。

**改善提案**:
- テキストで3回返信したら、AIが「次は音声で同じ内容を言ってみましょう🎤」と促す
- 音声フィードバックに「模範音声のスクリプト」を付加し、繰り返し練習を促す

---

### 3.4 自走期（Week 9+）: 継続と深化

**目標**: 自発的にBotを活用し、苦手分野を克服する

#### レベルアップの仕組み（新機能案）

現状、レベルは初回判定から変わらない。継続利用に伴うレベルアップ判定を追加。

```
判定条件（案）:
- 累計チャット50回以上
- 直近10回のAIフィードバックで「改善ポイント」が2つ以下の割合が70%以上
→ レベルアップ提案メッセージを送信
→ ユーザーが承認したらレベル変更
```

#### 苦手カテゴリの可視化（新機能案）

ウィークリーレポートに「カテゴリ別の正答傾向」を追加。

```
📊 今週の振り返り
テキスト: 12回
音声: 3回
学習日数: 5/7
ストリーク: 12日

📈 カテゴリ別:
  神社仏閣 ⭐⭐⭐⭐ 得意！
  食文化   ⭐⭐⭐  安定
  文化概念 ⭐⭐    もう少し！
  実践対応 ⭐     集中練習しましょう

[AIコメント]
```

### 4.5 スケーラビリティ設計（2名→数百名）

| 規模 | 課題 | 対策 |
|------|------|------|
| ~10名 | 管理は手動で可能 | 現状のまま運用 |
| 10-50名 | 個別フォローが困難に | 自動アラート（非アクティブ生徒の通知） |
| 50-100名 | コスト増大 | レート制限の動的調整、プラン別制限 |
| 100-300名 | バッチ処理の負荷 | Cloud Functions のメモリ/タイムアウト増設、並列バッチ |
| 300名+ | Firestore読み取り回数 | キャッシュ層追加、集計用コレクション |

#### Cloud Functions のスケーリング

```
現在:
  dailyPush: 300s timeout, 256MB memory, バッチ10件
  weeklyReport: 540s timeout, 512MB memory, バッチ10件

300名規模:
  dailyPush: 540s timeout, 512MB memory, バッチ20件
  weeklyReport: 分割実行（Cloud Tasks でキューイング）
  → 1回の呼び出しで50名ずつ処理、残りをCloud Tasksに委譲
```

#### LINE Messaging API のコスト試算

| 生徒数 | 月間メッセージ数 | LINE プラン | 月額 |
|--------|----------------|------------|------|
| 2名 | ~100通 | 無料（200通） | ¥0 |
| 20名 | ~700通 | ライト（5,000通） | ¥5,000 |
| 100名 | ~3,500通 | ライト（5,000通） | ¥5,000 |
| 300名 | ~10,500通 | スタンダード（30,000通） | ¥15,000 |

※ プッシュ通知1通/日 + レポート1通/週 + システムメッセージ ≒ 35通/月/人

## 5. 多言語対応設計（スペイン語版）

### 5.1 アーキテクチャ方針

**同一のLINE公式アカウントで複数言語に対応**するか、**言語別にアカウントを分ける**かの選択。

#### 推奨: 言語別LINE公式アカウント

| 方式 | メリット | デメリット |
|------|---------|----------|
| 同一アカウント | 管理が1つ、コード共有 | ユーザー体験が複雑、プロンプト混在 |
| **別アカウント** | **ユーザー体験がクリーン、独立運用** | アカウント2つの管理 |

→ 言語ごとにLINE公式アカウントを作成し、**バックエンドコードは共有**する。

### 5.2 コード上の多言語対応

```
src/
  prompts/
    systemPrompt.ts      → 言語パラメータを受け取るように変更
    voicePrompt.ts       → 同上
    reportPrompt.ts      → 同上
    pushMessages/
      en.ts              → 英語版質問プール
      es.ts              → スペイン語版質問プール
  config/
    constants.ts         → 言語別設定
    languages.ts         → 言語定義・UIテキスト
```

### 5.3 User型の拡張

```typescript
export interface User {
  // ... 既存フィールド ...
  language: "en" | "es";        // 学習対象言語
  uiLanguage: "ja";             // UI言語（当面は日本語固定）
  planId: string;               // 所属プランID
  courseId: string;              // 所属コースID
}
```

### 5.4 システムプロンプトの多言語化

```
英語版:
"You are an AI English coach specialized in tourism English
 for Japanese tour guides..."

スペイン語版:
"You are an AI Spanish coach specialized in tourism Spanish
 for Japanese tour guides. Your role is to help Japanese guides
 explain Japanese culture to Spanish-speaking tourists..."
```

**スペイン語版の特徴**:
- 比較対象がスペイン・中南米文化に変わる
  - 鳥居 → "Como el arco de entrada a una catedral"
  - おもてなし → "Como la hospitalidad mediterránea, pero anticipando necesidades"
- Speech-to-Text の言語設定を `es-ES` に変更
- 質問プールをスペイン語圏の観光客向けに調整

### 5.5 多言語展開のタイムライン

```
Phase 1（現在）: 英語版の運用・改善に集中
Phase 2（英語版安定後）: コードの多言語対応リファクタリング
Phase 3: スペイン語版の質問プール・プロンプト作成
Phase 4: スペイン語版LINE公式アカウント開設・テスト
Phase 5: スペイン語版リリース
```

## 5. レート制限と利用ガイドライン

### 現在の制限値

| 項目 | 制限 | 根拠 |
|------|------|------|
| テキスト | 20回/日 | 1回あたり約800トークン × 20 = API費用抑制 |
| 音声 | 5回/日 | Speech-to-Text + Claude のダブルAPI費用 |
| 音声長さ | 180秒/回 | Speech-to-Text の精度維持 |
| コンテキスト | 直近5件 or 60分 | トークン費用と文脈精度のバランス |

### 推奨利用パターン

```
【1日の理想的な利用】 所要時間: 10-15分

1. プッシュ通知の質問に英語で回答（テキスト）     ... 3-5分
2. AIフィードバックを読んで修正版を送信（テキスト） ... 2-3分
3. 修正版を音声で言ってみる（音声）              ... 2-3分
4. （余裕があれば）自分で考えた表現を送る         ... 3-5分

→ テキスト3-4回 + 音声1回 = 合計4-5回/日
→ 制限の25%程度しか使わないので十分な余裕
```

### スクール内でのルール案

1. **最低ライン**: 1日1回、プッシュ通知に返信する
2. **推奨ライン**: 上記パターンで1日10-15分
3. **ストリーク目標**: まず7日連続、次に30日連続を目指す
4. **音声の重要性**: 週に最低3回は音声メッセージで練習する

## 6. 講師の運用フロー

### 日常の確認作業

```
毎日（任意）:
  → Firestoreで生徒の最終アクティブ日を確認
  → 3日以上無活動の生徒に個別フォロー

毎週土曜（対面レッスン前）:
  → 各生徒のウィークリーレポートデータを確認
  → 「よく話したトピック」を参考にレッスン内容を調整
  → 音声/テキスト比率を確認 → 音声が少ない生徒には声がけ

月次:
  → 全生徒のレベル分布を確認
  → レベルアップ候補の生徒を識別
  → 問題プールの追加・入れ替え検討
```

### 対面レッスンとの連携方法

| タイミング | 講師のアクション |
|-----------|---------------|
| レッスン冒頭 | 「今週Botでやった質問の中で、一番難しかったのは？」 |
| ロールプレイ | Botで練習した表現を実際の会話で使わせる |
| レッスン終了時 | 「今週のBotのテーマは○○です。毎日1問やってみてください」 |
| 次週の準備 | 生徒のBot活動データを見てレッスン内容を調整 |

## 11. 技術的な課題と対策

### 11.1 コスト管理（スケール別）

| 項目 | 2名 | 20名 | 100名 | 300名 |
|------|-----|------|-------|-------|
| Claude API | ~$3/月 | ~$30/月 | ~$150/月 | ~$450/月 |
| Speech-to-Text | ~$0.4/月 | ~$3.6/月 | ~$18/月 | ~$54/月 |
| Firebase | 無料 | 無料 | ~$10/月 | ~$30/月 |
| LINE Messaging API | 無料 | ¥5,000/月 | ¥5,000/月 | ¥15,000/月 |
| **合計** | **~¥500** | **~¥10,000** | **~¥30,000** | **~¥85,000** |

※ Claude API: 5回/日/人, $0.01/回で試算
※ 300名規模では Claude Haiku への一部切り替えでコスト50%削減可能

#### コスト最適化戦略

| 規模 | 戦略 |
|------|------|
| ~50名 | 現状のまま（Claude Sonnet 4） |
| 50-100名 | プッシュ質問のAI生成をバッチ化（リアルタイム→事前生成） |
| 100名+ | 簡易フィードバック（定型パターン）はClaude Haiku、深いフィードバックはSonnet |
| 300名+ | レスポンスキャッシュ（類似質問への回答を再利用） |

### 11.2 データプライバシー

- チャットログにはガイド実務の話題が含まれうる
- Firestore Security Rulesで生徒同士のデータアクセスを防止
- 講師のみ担当コースの生徒データを閲覧可能とする
- GDPR/個人情報保護法に対応するデータ削除機能（退会時）
