# 運用設計書 レビュー

4つのシニア職種の観点から `operation-design.md` をレビューする。

---

## A. シニアマーケティングマネージャーの視点

### 評価: 良い点

- **"comparison, not translation" は強力なブランドメッセージ**。他の英語学習サービスとの明確な差別化になっている。通訳ガイドに特化した唯一のAI英語コーチという位置づけは市場で独自性がある。
- ユーザージャーニーの4フェーズ設計（導入→習慣形成→成長→自走）は、マーケティングファネルとしてもきれいに対応している。

### 指摘事項

#### MA-1: ターゲットペルソナの解像度が低い（重要度: 高）

設計書全体を通じて「通訳ガイド向け」としか書かれていないが、実際には大きな差異があるはず。

- **既に全国通訳案内士の資格を持っている人** vs **これから取得を目指す人**
- **フルタイムガイド** vs **副業・週末ガイド**
- **英語での実務経験がある人** vs **まだ現場に出ていない人**
- **年齢層**: 20-30代 vs 40-60代（デジタルリテラシーに大きな差）

ペルソナを2-3パターン定義し、それぞれに対するメッセージングと導入チャネルを設計すべき。

#### MA-2: 集客・新規獲得の設計が完全に欠落（重要度: 高）

2名→数百名のスケール計画があるが、**どうやって新規生徒を獲得するかの戦略がゼロ**。

追加すべき項目:
- **集客チャネル**: SNS（Instagram, X）、通訳ガイド関連コミュニティ、資格学校との提携、口コミ紹介制度
- **無料体験の設計**: 何日間無料か、体験後のコンバージョン導線
- **紹介プログラム**: 既存生徒が新規を紹介するインセンティブ設計
- **コンテンツマーケティング**: Bot の質問・回答例をSNSで公開し認知拡大

#### MA-3: 料金体系・課金モデルの記載がない（重要度: 高）

コスト試算（サプライサイド）はあるが、**生徒からいくら頂くのか**（デマンドサイド）の設計がない。

検討すべきモデル:
- 対面レッスン料金にBot利用料を含める（バンドル型）
- Bot単体のサブスクリプション（月額制）
- フリーミアム（基本機能無料、プレミアム機能有料）

これがないと、スケール時の収益性が判断できない。

#### MA-4: ブランド名・トンマナの統一指針がない（重要度: 中）

「AI English Coach」という名前で十分か？スペイン語版を出すときに「AI Spanish Coach」とするのか？統一ブランド名（例: "GuideCoach AI"）を検討すべき。

Botのキャラクター設定（「supportive senpai」）は良いが、もう少し具体的なブランドパーソナリティガイドラインがあると、プロンプト改善やマーケティング素材の一貫性が保てる。

## B. シニアカスタマーサクセスの視点

### 評価: 良い点

- オンボーディングの3ステップ設計は丁寧で、「初日に1回返信させる」という明確なアクティベーション目標が良い。
- ナッジメッセージのタイプ分け（gentle / strong / streak_boost）はリエンゲージメント設計として適切。
- ウィークリーレポートは生徒のセルフモニタリングに有効。

### 指摘事項

#### CS-1: チャーン（離脱）防止の設計が弱い（重要度: 高）

ナッジメッセージ以外に**離脱予兆を検知し介入するプロセス**がない。

追加すべき仕組み:
- **離脱リスクスコアリング**: 利用頻度の減少傾向、応答率の低下、ストリーク切断パターンを複合的に判定
- **段階的介入フロー**:
  - 2日無活動 → Bot自動ナッジ（現在の仕組み）
  - 5日無活動 → 講師にアラート通知（LINEで「山田さんが5日無活動です」）
  - 10日無活動 → 講師から直接メッセージ or 電話
  - 30日無活動 → 休会提案 or 退会フォロー
- **離脱理由の収集**: ブロック/unfollow時にアンケート（LINE上では技術的に困難なため、対面レッスン時にヒアリング）

#### CS-2: 生徒の「成功体験」の設計が不十分（重要度: 高）

レベルアップ判定（累計50回 + 改善ポイント70%以下）はあるが、**短期的な達成感**が足りない。

追加すべき仕組み:
- **マイクロ成功体験**: 初回返信完了、3日連続、7日連続、30日連続のマイルストーン通知
- **バッジ/称号システム**: 「神社仏閣マスター」「音声練習の達人」などカテゴリ別の達成バッジ
- **Before/After の可視化**: 初期の回答と最近の回答を並べて見せ、成長を実感させる
- **月次サマリー**: ウィークリーだけでなく月次で「先月からこれだけ成長しました」

#### CS-3: 生徒間のコミュニティ/相互学習の要素がゼロ（重要度: 中）

完全に1対Bot の設計になっているが、学習継続の最大の動力は**仲間の存在**。

検討すべき施策:
- **匿名ランキング**: 「今週の学習時間ランキング」を全体に配信（プライバシー配慮で匿名）
- **今週のベストアンサー**: 優秀な回答を講師が選んで全体共有
- **グループチャレンジ**: 「クラス全員で合計100回を目指そう」

#### CS-4: フィードバックの質に対する満足度測定がない（重要度: 中）

AIのフィードバックが本当に役に立っているかの確認手段がない。

- 月に1回、ランダムなフィードバック後に「この回答は参考になりましたか？ 👍 / 👎」をLINEのクイックリプライで取得
- 低評価が続いたら、プロンプトの改善やレベル再判定のトリガーにする

## C. シニアセールスマネージャーの視点

### 評価: 良い点

- 通訳ガイドに特化というニッチ市場は、セールス上は「専門性の高さ」として訴求しやすい。
- 516回以上のツアー実績を持つ講師のノウハウがAIに組み込まれているのは強い信頼材料。

### 指摘事項

#### SA-1: 商品パッケージングと価格設計がない（重要度: 最高）

**今の設計書では「何を売るのか」が明確でない。** 営業活動ができない。

提案するパッケージ構成:
```
┌───────────────────────────────────────────┐
│  プラン名          │ 内容            │ 月額   │
├───────────────────────────────────────────┤
│  Bot Only         │ AI Bot利用のみ   │ ¥3,000 │
│  Standard         │ Bot + 月2回対面  │ ¥15,000│
│  Premium          │ Bot + 週1回対面  │ ¥30,000│
│  Enterprise       │ 法人向けカスタム │ 要相談  │
└───────────────────────────────────────────┘
```

#### SA-2: 法人（B2B）チャネルの検討がない（重要度: 高）

数百名規模に最も早く到達する方法は**個人営業の積み上げではなく法人契約**。

ターゲット法人:
- **旅行会社（JTB, HIS等）**: インバウンド部門のガイド研修ツールとして
- **通訳ガイド養成学校**: カリキュラムの一部として採用
- **地方自治体**: 観光ボランティアガイドの育成プログラム
- **ホテルチェーン**: コンシェルジュ/フロントスタッフの英語研修

法人契約のメリット:
- 一度に10-50名単位で獲得可能
- 継続率が個人より高い（組織の意思決定で導入するため）
- 単価を上げやすい（研修費として予算化される）

#### SA-3: 競合分析・差別化ポイントの整理がない（重要度: 高）

営業時に必ず聞かれる「他のサービスとの違い」に対する回答が設計書にない。

主な競合:
- **汎用AI英会話アプリ** (Speak, ELSA等): 通訳ガイド特化ではない
- **英会話スクール** (Berlitz, ECC等): AI活用がない、高額
- **通訳ガイド試験対策講座**: 試験対策のみで実務スキルは弱い
- **ChatGPT直接利用**: プロンプト設計がない、ガイド実務知識がない

差別化マトリクスを設計書に追加すべき。

#### SA-4: 成果保証・トライアル設計がない（重要度: 中）

「使ってみないとわからない」ものを売るには、リスクリバーサルが必要。

- **7日間無料トライアル**: プッシュ通知＋テキスト3回/日＋音声1回/日
- **成果保証**: 「3ヶ月で○○ができるようになる」（例: 5つの日本文化概念を英語で説明できるようになる）
- **事例・実績**: 現在の2名の生徒のBefore/After事例を蓄積

## D. シニアテクノロジーマネージャーの視点

### 評価: 良い点

- Firebase + Cloud Functions のサーバーレス構成は小規模からスタートするのに適切。
- Anthropic Claude の選択は品質面で正しい。Speech-to-Text の Google Cloud 利用も安定性が高い。
- スケーリングの段階別設計（10名→50名→100名→300名+）は現実的。

### 指摘事項

#### TM-1: 監視・アラートの設計がない（重要度: 高）

本番運用するにあたり、**システムが正常に動いているか**を知る手段が設計されていない。

必要な監視項目:
- **API可用性**: Claude API, Google Speech-to-Text, LINE Messaging API のエラー率
- **レスポンスタイム**: webhook の応答時間（30秒超で LINE が再送する）
- **レート制限到達**: Claude API の429エラー頻度
- **コスト監視**: 日次/週次のAPI利用量アラート（急増検知）
- **バッチ処理**: dailyPush / weeklyReport の成功率・処理時間

推奨ツール:
- Cloud Monitoring + Cloud Alerting（GCP標準）
- 重要エラーはLINEで講師に通知（Bot自身のエラーをBot自身で通知は危険なので、メール or Slack併用）

#### TM-2: テスト戦略がない（重要度: 高）

現在のコードベースにテストが見当たらない。数百名規模で運用するなら最低限必要:

- **単体テスト**: プロンプト生成、レベル判定、ストリーク計算、レート制限ロジック
- **統合テスト**: LINE webhook → Firestore → Claude API のモック付きE2Eテスト
- **負荷テスト**: 300名同時刻のdailyPush実行シミュレーション
- **プロンプト回帰テスト**: プロンプト変更時に出力品質が劣化していないかの自動チェック

#### TM-3: 環境分離（本番/ステージング）がない（重要度: 高）

現状、本番環境1つのみ。プロンプト変更やコード変更が直接生徒に影響する。

必要な環境:
- **開発環境**: ローカル + Firebase Emulators
- **ステージング環境**: 本番と同構成の別プロジェクト（講師自身がテスト生徒としてBotを使う）
- **本番環境**: 実際の生徒が使う

#### TM-4: LINE公式アカウントの分離ではなく環境変数での言語切替を推奨（重要度: 中）

設計書では言語別にLINE公式アカウントを分けることを推奨しているが、以下の理由で**1アカウント + ユーザー属性での言語切替**も検討すべき:

- LINE公式アカウント2つ = Webhook URL 2つ = Cloud Functions 2つのデプロイ管理
- 同一生徒が英語とスペイン語の両方を学びたい場合に対応できない
- バックエンドコード共有なら、`User.language` で切り替える方が実装もシンプル

ただし、この判断はマーケティング観点（ブランドの見え方）とのトレードオフなので、最終判断はビジネス側と合議すべき。

#### TM-5: データバックアップ・災害復旧計画がない（重要度: 中）

Firestore のデータ（生徒の全学習履歴）を失うとサービス継続が困難。

- Firestore の定期バックアップ設定（日次）
- バックアップからのリストア手順の文書化
- リージョン障害時の対応方針（asia-northeast1 単一リージョン）

#### TM-6: AIのハルシネーション対策が未記載（重要度: 中）

通訳ガイド向けという性質上、**文化的に不正確な情報を生徒に教えるリスク**がある。

- 文化・歴史的事実に関するフィードバックの精度検証プロセス
- 明らかに不正確な回答を生徒が報告する仕組み（「この回答は間違っています」コマンド）
- プロンプトに「確信がない場合は断定を避ける」指示を追加

## 総合評価サマリー

| 観点 | 現状の成熟度 | 最優先の改善点 |
|------|------------|-------------|
| マーケティング | ★★☆☆☆ | 集客戦略・料金体系・ペルソナ定義 |
| カスタマーサクセス | ★★★☆☆ | チャーン防止の自動化、短期的成功体験 |
| セールス | ★☆☆☆☆ | 商品パッケージング、法人チャネル開拓 |
| テクノロジー | ★★★☆☆ | 監視/テスト/環境分離 |

### フェーズ別の追加タスク提案

#### Phase 1 に追加すべき項目（今すぐ）
1. ペルソナ2-3パターンの定義
2. 料金体系の仮設定
3. 基本的な監視・アラート設定
4. 7日間無料トライアルの設計
5. 離脱検知→講師アラートの自動化

#### Phase 2 に追加すべき項目（10-50名）
6. 紹介プログラムの導入
7. マイルストーン/バッジシステム
8. 単体テストの整備
9. ステージング環境の構築
10. 法人営業資料の作成

#### Phase 3 に追加すべき項目（50名以上）
11. コンテンツマーケティング戦略
12. B2B向けエンタープライズプラン
13. 負荷テスト・プロンプト回帰テスト
14. データバックアップの自動化
15. 生徒コミュニティ機能
