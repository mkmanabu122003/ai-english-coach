# 継続利用改善 要件定義書

## 背景・目的

語学学習において最も重要なのは継続利用である。現状のBotは「使い始める仕組み」は優れているが、「離れかけた人を引き戻す仕組み」と「日々の小さな成長を実感させる仕組み」が不足している。

本改善は、ユーザーの継続率（Day 7 / Day 30 リテンション）向上を目的とする。

## 改善施策一覧

| # | 施策名 | 優先度 | 想定効果 | 工数目安 |
|---|--------|--------|---------|---------|
| 1 | オンボーディング段階的ナビゲーション | P0 | Day 3 リテンション向上 | S |
| 2 | ストリーク切れリカバリー | P0 | Day 7+ リテンション向上 | S |
| 3 | Freeユーザー向け簡易週次レポート | P1 | Free→Pro転換率向上 | S |
| 4 | 離脱ユーザーへの自動リエンゲージメント | P1 | 休眠復帰率向上 | M |
| 5 | プッシュ通知の学習済みスキップ改善 | P2 | 習慣形成の安定化 | S |
| 6 | Free プラン Try again 回数カウント緩和 | P2 | Free 体験の満足度向上 | S |
| 7 | 未回答プッシュ問題のリマインド | P2 | プッシュ通知への反応率向上 | M |

工数目安: S = 1-2日, M = 3-5日, L = 1週間以上

## 施策 1: オンボーディング段階的ナビゲーション

### 現状の課題

`onboardingStatus`（`firstText`, `levelSet`, `pushTimeSet`, `firstVoice`, `streak3`）のデータは記録しているが、次のステップへ誘導するメッセージが存在しない。ウェルカムメッセージ→初回チャット→ヒント表示で導線が途切れる。

語学アプリの離脱は大半が最初の3日以内に起きるため、この期間にユーザーを段階的に導くことが最重要。

### 要件

#### 1-1. ステップ完了時の次アクション誘導メッセージ

各オンボーディングステップの完了を検知したタイミングで、AI応答の末尾に次ステップへの誘導メッセージを追加する。

| ステップ完了 | 追加メッセージ |
|-------------|-------------|
| `firstText` + `levelSet` | 「レベルが判定されました！\n毎日練習問題を届けます。時間を変えたい場合は「通知設定 HH:MM」と送ってください。」 |
| `pushTimeSet` | 「通知設定が完了しました！\n次は音声メッセージも試してみましょう🎤 同じ英文を声に出すと定着率が上がります。」 |
| `firstVoice` | 「音声練習もクリア！テキストと音声を組み合わせると効果的です。\nまずは3日連続を目指してみましょう 🔥」 |
| `streak3` | 「3日連続達成！習慣化の第一歩です。\n次は1週間連続を目指しましょう。ここからが本番です！」 |

#### 1-2. 変更対象ファイル

- `src/handlers/textChat.ts` — `onboardingStatus`更新ロジックの後に誘導メッセージ追加
- `src/handlers/voiceChat.ts` — 同上
- `src/config/languages.ts` — 誘導メッセージの文字列定義を追加（`onboardingMessages`）

#### 1-3. 仕様詳細

- 誘導メッセージは `\n\n──────\n` 区切りで応答末尾に追加（既存の `firstChatHint` と同じ形式）
- マイルストーンメッセージと同時に発生した場合、マイルストーン→オンボーディング誘導の順で表示
- 各ステップの誘導メッセージは1回のみ表示（`onboardingStatus`の該当フラグが `false` → `true` に変わったタイミングのみ）
- `pushTimeSet`は`通知設定 HH:MM`コマンド実行時にフラグ更新＋誘導メッセージを返す

#### 1-4. 受け入れ基準

- [ ] 新規ユーザーが初回テキストを送信→レベル判定後に通知設定の案内が表示される
- [ ] `通知設定 21:00` を送信→音声練習の案内が表示される
- [ ] 初回音声を送信→3日連続チャレンジの案内が表示される
- [ ] 3日連続達成→1週間チャレンジの案内が表示される
- [ ] 各メッセージが2回以上表示されない
- [ ] 英語Bot・スペイン語Botの両方で動作する

## 施策 2: ストリーク切れリカバリー

### 現状の課題

ストリークが途切れると`currentStreak`が0にリセットされるが、ユーザーへのフォローがない。`longestStreak`は記録されているが`レベル確認`コマンドにも表示されない。7日連続の後にストリークが切れた場合の喪失感は大きく、そのまま離脱するパターンが語学アプリでは非常に多い。

### 要件

#### 2-1. ストリーク切れ後の復帰メッセージ

ストリークが途切れた後の最初のチャットで、励ましメッセージを表示する。

**発動条件:**
- `currentStreak` が 0 である（前回の学習から2日以上経過）
- `longestStreak` が 3 以上である（ある程度の実績がある）
- 直前のチャットで `currentStreak` が 0 → 1 に変わった

**メッセージ:**
```
──────
💪 おかえりなさい！最長記録は{longestStreak}日です。
また1日目からコツコツ積み上げましょう！
```

#### 2-2. `レベル確認`コマンドに最長ストリーク表示を追加

**現状:**
```
レベル: intermediate
連続学習: 5日
累計チャット: 23回
```

**改善後:**
```
レベル: intermediate
連続学習: 5日（最長: 14日）
累計チャット: 23回
```

#### 2-3. 変更対象ファイル

- `src/handlers/textChat.ts` — ストリーク復帰検知＋メッセージ追加
- `src/handlers/voiceChat.ts` — 同上
- `src/config/languages.ts` — 復帰メッセージ文字列追加、`levelCheckReply`に`longestStreak`引数追加

#### 2-4. 受け入れ基準

- [ ] 7日連続学習→2日休み→再開時に「おかえりなさい！最長記録は7日です」と表示される
- [ ] ストリーク0の初心者（longestStreak < 3）には復帰メッセージが表示されない
- [ ] `レベル確認`コマンドで最長ストリークが表示される
- [ ] 復帰メッセージは復帰後の最初の1通のみ表示される

## 施策 3: Freeユーザー向け簡易週次レポート

### 現状の課題

週次レポートがProユーザー限定（`scheduler.ts:134`の`plan !== "free"`フィルタ）。Freeユーザーは自分の学習進捗を振り返る手段がなく、成長実感が得にくい。またProプランの価値を体感する機会も失われている。

### 要件

#### 3-1. Freeユーザー向け簡易レポートの配信

Freeユーザーにも週次レポートを配信する。ただしAI生成コメントは含めず、統計サマリーのみとする。

**Freeユーザー向けレポート:**
```
📊 今週の振り返り
テキスト: 8回
音声: 0回
学習日数: 4/7
ストリーク: 4日

💡 Proプランでは毎週あなた専用のアドバイスが届きます。
```

**Proユーザー向けレポート（変更なし）:**
```
📊 今週の振り返り
テキスト: 12回
音声: 3回
学習日数: 5/7
ストリーク: 14日

[AIが生成する個別コメント]
```

#### 3-2. 変更対象ファイル

- `src/handlers/scheduler.ts` — `weeklyReport`関数のフィルタ条件変更、Free/Pro分岐追加
- `src/config/languages.ts` — Proアップセル文言の追加（`weeklyReportFreeFooter`）

#### 3-3. 仕様詳細

- Freeユーザーのレポートでは `chatCompletion` APIを呼ばない（コスト削減）
- Freeレポートも `weeklyReports` コレクションに保存する（分析用）
- 当該週にチャット0回のユーザーにもレポートを送る（「今週はお休みでしたね」的な位置づけ）

#### 3-4. 受け入れ基準

- [ ] Freeユーザーに毎週日曜10:00にレポートが届く
- [ ] FreeレポートにはAI生成コメントが含まれず、代わりにProアップセル文言がある
- [ ] FreeレポートではClaude APIが呼ばれない
- [ ] Proユーザーのレポートは既存仕様から変更なし

## 施策 4: 離脱ユーザーへの自動リエンゲージメント

### 現状の課題

現在のナッジは3段階（streak_boost / gentle_nudge / strong_nudge）だが、`strong_nudge`（3日以上不活動）が最大で、それ以上の長期離脱者への対策がない。`churnDetection`は講師へのアラートのみで、ユーザー本人への自動介入がない。Freeユーザーの離脱は完全に放置されている。

### 要件

#### 4-1. 第4段階ナッジの追加（復帰促進）

| ナッジ種別 | 条件 | 目的 |
|-----------|------|------|
| `streak_boost` | 昨日アクティブ＆ストリーク継続 | 継続の後押し |
| `gentle_nudge` | 1-2日不活動 | 軽い促し |
| `strong_nudge` | 3-6日不活動 | 強めの促し |
| **`comeback`（新規）** | **7-13日不活動** | **復帰への呼びかけ** |

**comeback メッセージ例（各言語2パターン以上）:**
```
"しばらくお会いしていませんね。1分でできる簡単な問題を用意しました。気軽にどうぞ👋"
"英語の勘を取り戻しましょう！簡単な問題から再スタートできます🔄"
```

#### 4-2. comebackナッジ時の問題レベル引き下げ

`comeback`ナッジの場合、ユーザーの現在レベルに関わらず**1段階下のレベル**の問題を出題する。心理的ハードルを下げ、「できた」体験から再開してもらう狙い。

| ユーザーレベル | comeback時の出題レベル |
|-------------|---------------------|
| advanced | intermediate |
| intermediate | beginner |
| beginner | beginner（変更なし） |

#### 4-3. 14日以上不活動ユーザーへのデイリープッシュ停止

14日以上不活動のユーザーにはデイリープッシュを停止する。過剰な通知はブロック（unfollow）に繋がるため。

**停止条件:** `lastActiveDate`が14日以上前

**代替施策:** 14日以上不活動ユーザーには**週1回（月曜のみ）** に限定した復帰メッセージを送る。

#### 4-4. 変更対象ファイル

- `src/prompts/pushMessages.ts` — `NudgeType`に`comeback`追加、メッセージ定義追加
- `src/handlers/scheduler.ts` — `determineNudgeType`にcomeback判定追加、14日超の除外ロジック追加、comeback時のレベル引き下げロジック追加
- `src/config/languages.ts` — `nudgeMessages`に`comeback`メッセージ追加

#### 4-5. 受け入れ基準

- [ ] 7日不活動のユーザーにcomebackナッジ＋レベル引き下げ問題が届く
- [ ] 14日以上不活動のユーザーにはデイリープッシュが停止される
- [ ] 14日以上不活動のユーザーに週1回（月曜）のみ復帰メッセージが届く
- [ ] 既存の3段階ナッジ（streak_boost / gentle / strong）の挙動に変更がない

## 施策 5: プッシュ通知の学習済みスキップ改善

### 現状の課題

`scheduler.ts:55`で`lastActiveDate === todayJST`の場合プッシュ通知をスキップしている。通知の有無が日によって変わると、ハビットトリガーとしての信頼性が下がる。

### 要件

#### 5-1. 学習済みユーザーへの励ましメッセージ送信

当日すでに学習済みのユーザーには、問題の代わりに短い励ましメッセージを送る。

**メッセージ例:**
```
"今日も練習おつかれさまでした！{currentStreak}日連続です🔥 明日も頑張りましょう！"
"今日の学習クリア！この調子で続けていきましょう 💪"
```

#### 5-2. 設定による制御

この機能はユーザー設定で無効化できるようにはしない（シンプルさ優先）。ただし、将来的に設定追加する余地は残す。

#### 5-3. 変更対象ファイル

- `src/handlers/scheduler.ts` — `sendDailyPush`のスキップロジックを励ましメッセージ送信に変更
- `src/config/languages.ts` — 励ましメッセージ文字列追加（`dailyCompletedMessages`）

#### 5-4. 受け入れ基準

- [ ] 当日学習済みユーザーにプッシュ通知時刻に励ましメッセージが届く
- [ ] 励ましメッセージにはストリーク日数が含まれる
- [ ] 未学習ユーザーへのプッシュ通知（既存）に変更がない
- [ ] 励ましメッセージは `interventions` に記録されない（統計の歪み防止）

## 施策 6: Free プラン Try again 回数カウント緩和

### 現状の課題

Freeプラン（3回/日）では、添削→Try again→再添削で1往復2回を消費する。実質1.5往復で制限に達し、「学習をやりきった」感覚が得にくい。アップセルの価値を感じる前に制限にぶつかる。

### 要件

#### 6-1. Try again 応答後60秒以内の再送信をカウントしない

AIの応答に `🔄` （Try again）が含まれている場合、その応答から**60秒以内**のユーザーの次の1通をレート制限のカウントから除外する。

**条件:**
- 直前のAI応答に `🔄` が含まれている
- 直前のAI応答からの経過時間が60秒以内
- Freeプランのユーザーのみ（Proは十分な回数があるため対象外）
- 1回のTry againにつき1通のみ免除

#### 6-2. 変更対象ファイル

- `src/handlers/textChat.ts` — Try again判定＋カウント免除ロジック追加
- `src/services/firestore.ts` — 直前のchatLogのAI応答とタイムスタンプ取得に使用（既存の`getRecentChatLogs`を活用）
- `src/middleware/rateLimiter.ts` — `checkRateLimit`にTry again免除フラグ引数を追加

#### 6-3. 受け入れ基準

- [ ] Freeユーザーが添削を受けた後、60秒以内に再送信→カウントされない（実質4回使える）
- [ ] 60秒を超えた再送信は通常通りカウントされる
- [ ] Try againなしの応答（文法質問への回答など）後の送信は通常通りカウントされる
- [ ] Proユーザーの挙動に変更がない
- [ ] 免除は連続適用されない（Try again免除→再度Try again→2回目は免除されない）

## 施策 7: 未回答プッシュ問題のリマインド

### 現状の課題

デイリープッシュで問題が送られても、回答しなかった場合のフォローがない。自由入力型は初心者にとって「何を送ればいいかわからない」で止まりやすい。

### 要件

#### 7-1. 前日の未回答問題をナッジに含める

前日にプッシュ問題を送ったが、その日にチャットがなかった場合、翌日のナッジメッセージに「昨日の問題」を再掲する。

**メッセージ構成:**
```
[ナッジメッセージ]

📌 昨日の問題にまだ答えていません：
「{前日の問題テキスト}」

もちろん、別の話題でもOKです！
```

#### 7-2. データ設計

`users`ドキュメントに以下フィールドを追加:

```typescript
lastPushQuestion?: {
  questionText: string;
  sentAt: Timestamp;
  answered: boolean;
}
```

- プッシュ送信時に `lastPushQuestion` を更新（`answered: false`）
- ユーザーがチャットを送信した場合、`answered: true` に更新
- 翌日のプッシュ時に `answered === false` なら再掲

#### 7-3. 変更対象ファイル

- `src/types/index.ts` — `User`型に`lastPushQuestion`フィールド追加
- `src/handlers/scheduler.ts` — プッシュ送信時に`lastPushQuestion`記録、翌日のプッシュ時に未回答チェック＋再掲
- `src/handlers/textChat.ts` — チャット受信時に`lastPushQuestion.answered`を`true`に更新
- `src/handlers/voiceChat.ts` — 同上
- `src/config/languages.ts` — リマインドメッセージテンプレート追加

#### 7-4. 受け入れ基準

- [ ] プッシュ問題を受け取って回答しなかった場合、翌日のプッシュに前日の問題が再掲される
- [ ] プッシュ問題に回答した場合、翌日は通常通り新しい問題が出る
- [ ] 再掲は1回のみ（2日連続で同じ問題が再掲されない）
- [ ] 再掲された問題は `recentQuestions` にすでに含まれているため、通常の新規問題選択からは除外済み

## 実装順序の推奨

```
Phase 1 (P0): 施策1 + 施策2
  → 既存データ構造を活用。最小工数で最大効果。
  → Day 3 / Day 7 リテンションへの直接的な改善。

Phase 2 (P1): 施策3 + 施策4
  → 週次レポートのFree展開でPro転換を促進。
  → 長期離脱者への自動介入でスケーラブルな運用へ。

Phase 3 (P2): 施策5 + 施策6 + 施策7
  → 体験の細部を磨く改善。
  → Phase 1-2の効果測定後に優先度を再評価。
```

## 効果測定

各施策の効果は以下のKPIで測定する。既存の `stats/daily` コレクションと `users` コレクションのデータで算出可能。

| KPI | 測定方法 | 対象施策 |
|-----|---------|---------|
| Day 3 リテンション | 登録3日後にアクティブなユーザーの割合 | 施策1 |
| Day 7 リテンション | 登録7日後にアクティブなユーザーの割合 | 施策1, 2 |
| Day 30 リテンション | 登録30日後にアクティブなユーザーの割合 | 施策2, 4 |
| ストリーク切れ後の復帰率 | ストリーク0になった後7日以内に再開した割合 | 施策2 |
| プッシュ通知反応率 | プッシュ送信後24時間以内にチャットしたユーザーの割合 | 施策5, 7 |
| Free→Pro転換率 | Freeユーザーがbot_proに変更した割合 | 施策3, 6 |
| 週間アクティブ率 | 週に1回以上チャットしたユーザーの割合 | 全施策 |
| ブロック率 | unfollowイベント数 / 全ユーザー数 | 施策4, 5（悪化していないか監視） |
